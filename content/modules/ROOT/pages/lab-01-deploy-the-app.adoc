= Lab 1: Deploy the App -- 3 Ways to Get Code Running
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application onto OpenShift -- and this lab proves it.
You'll deploy the DataSphere application three different ways: from a pre-built container image,
from source code using Source-to-Image, and as a complete multi-component Helm chart. Along
the way you'll learn the three building blocks that make every OpenShift application work: *Pods*,
*Services*, and *Routes*.

By the end, you'll have a fully working DataSphere instance -- UI, API, and database --
deployed from a single Helm chart, and you'll understand exactly what each piece does.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application from a pre-built image using the `oc` CLI
* Build and deploy an application from source code using Source-to-Image (S2I)
* Deploy a multi-component application in a single command using a Helm chart
* Explain when to reach for each of the three deployment methods

=== Step 1.3: Explore the Pod

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives
them a shared network and storage context.

List the pods in your project:

[source,role="execute"]
----
oc get pods
----

[%collapsible]
.Expected output
====
----

datasphere-redis-69c8864976-xxxxx   1/1     Running   0          5m
datasphere-ui-597f67c7bc-xxxxx      1/1     Running   0          5m
----
====

A *Deployment* manages this lifecycle and ensures the right number of pods are always running.

=== Step 1.4: Explore the Service

A *Service* gives your application a stable internal network address. Even if the Pod restarts
and gets a new IP address, the Service address stays the same.

[source,role="execute"]
----
oc get service datasphere-ui
----

[%collapsible]
.Expected output
====
----
NAME            TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
datasphere-ui   ClusterIP   172.30.198.47   <none>        8080/TCP   5m
----
====

[source,role="execute"]
----
oc describe service datasphere-ui
----

[%collapsible]
.Expected output (key fields)
====
----
Name:              datasphere-ui
Selector:          app=datasphere,component=ui
Type:              ClusterIP
Port:              8080/TCP
Endpoints:         10.x.x.x:8080
----
====

Notice the `Selector` field: it uses labels (`app=datasphere,component=ui`) to identify which pods to
route traffic to. This label-based selection is how OpenShift connects resources together.

=== Step 1.5: Explore the Route

A *Route* exposes a Service to traffic from outside the cluster. It provides a public hostname
and handles TLS termination so your users can access the app over HTTPS.

[source,role="execute"]
----
oc describe route datasphere-ui
----

[%collapsible]
.Expected output (key fields)
====
----
Name:            datasphere-ui
Host:            datasphere-ui-{user}-datasphere.apps.{cluster_subdomain}
TLS Termination: edge
Insecure Policy: Redirect
Service:         datasphere-ui
Port:            8080
----
====

Notice the `Host` field: that's the public URL you opened in your browser.
Notice the `TLS` section: OpenShift handles the certificate automatically.

=== Step 1.6: Why datasphere-api has no Route

Look back at the `oc get all` output from Step 1.2. `datasphere-api` has a Service but no Route.
That's deliberate: the API only needs to be reachable from within the cluster, not from the internet.

[source,role="execute"]
----
oc get service datasphere-api
----

[%collapsible]
.Expected output
====
----
NAME             TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)    AGE
datasphere-api   ClusterIP   172.30.14.22   <none>        8080/TCP   5m
----
====

The UI talks to the API by Service name -- `datasphere-api` -- and OpenShift's internal DNS
resolves it automatically. No Route needed.

=== How they connect

[mermaid]
....
flowchart TB
    B["Browser"]
    R["Route\n(datasphere-ui)"]
    S["Service\n(stable internal IP)"]
    P1["Pod\n(datasphere-ui-n7rlv)"]
    P2["Pod\n(datasphere-ui-fgh12)"]:::dashed
    P3["Pod\n(datasphere-ui-zxv9k)"]:::dashed

    B -->|"HTTPS"| R
    R --> S
    S -->|"label selector"| P1
    S -->|"label selector"| P2
    S -->|"label selector"| P3

    classDef dashed stroke-dasharray: 5 5
....

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to via label selectors.
The Pod runs your container.

The dashed pods aren't running right now, but if you scaled up, the Service would
automatically route traffic to every pod with a matching label.

== Section 2: Deploy it yourself -- two different ways (14 minutes)

Now that you understand the building blocks, it's time to deploy DataSphere yourself.
You'll use two different methods: first from a pre-built container image, then from
source code using Source-to-Image. Each method has its place, and by the end you'll
know when to reach for which.

TIP: If you're already familiar with deploying container images and S2I builds on OpenShift,

----

----

This project is empty -- a clean workspace for you to build in.

=== Deploy the UI from a container image

The fastest way to deploy an application is from an image that's already built and published
to a registry. You just point OpenShift at the image and it handles the rest.

=== Step 2.2: Deploy the application

Deploy the DataSphere UI from its container image:

[source,role="execute",subs="attributes+"]
----
oc new-app {datasphere_ui_image} --name=datasphere-ui
----

[%collapsible]
.Expected output
====
----
--> Found container image ... ({datasphere_ui_image})
    ...
--> Creating resources ...
    imagestream.image.openshift.io "datasphere-ui" created
    deployment.apps "datasphere-ui" created
    service "datasphere-ui" created
--> Success

     'oc expose service/datasphere-ui'
    Run 'oc status' to view your app.
----
====

`oc new-app` created three resources: an ImageStream to track the image version, a Deployment to
manage your pods, and a Service for internal network access. The output even tells you the next

no downtime, no manual coordination.

You can run `oc get pods` to see your new pod running in this project.

=== Deploy the API via Source-to-Image

Rather than deploying from a pre-built image, you can point OpenShift directly at a source
code repository. OpenShift detects the language, installs dependencies, builds a container
image, and deploys it -- all inside the cluster, with no Dockerfile required.
This is *Source-to-Image* (S2I).

=== Step 2.7: Build and deploy the API from source

The `python~` prefix tells OpenShift two things: use the Python builder image, and build from
the source that follows. OpenShift clones the repository, detects `requirements.txt`,
installs dependencies, and packages everything into a container image -- all inside the cluster,
with no Dockerfile required.

[source,role="execute",subs="attributes+"]
----
oc new-app python~{gitea_url}/{user}/datasphere-api --name=datasphere-api
----

[source,role="execute"]

== Section 3: Deploy everything with a Helm chart (6 minutes)

You deployed DataSphere in two stages: the UI from a container image, the API from source.
That works, but in practice most teams don't run individual `oc new-app` commands one at a time.
They package their application as a *Helm chart* -- a single installable unit that deploys
every component together.

A chart is a directory of templated Kubernetes manifests plus a `values.yaml` file that controls
configurable inputs (image tags, theme, replica count, and so on). When you run `helm install`,
Helm renders those templates with your chosen values and submits all the resulting manifests to
the cluster in one shot. It also records what it installed as a named *release*, so it can
upgrade, roll back, or cleanly remove every resource it owns.

=== Step 3.2: Deploy the full application

Install the DataSphere chart -- this deploys UI, API, and Redis in one command:

[source,role="execute",subs="attributes+"]
----
helm install datasphere {gitea_url}/gitea-admin/datasphere-gitops/archive/main.tar.gz
----

The first argument after `helm install` is the *release name* (`datasphere`). Helm uses it to
track this installation. The second argument is the chart location -- a Helm chart stored in the
Gitea repository as a Git archive.

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-lab
STATUS: deployed
REVISION: 1
DESCRIPTION: Install complete
TEST SUITE: None
----
====

`REVISION: 1` tells you this is the first install. Each `helm upgrade` increments the revision
and Helm keeps the history -- you can roll back to any previous revision if something goes wrong.

[source,role="execute"]
----

----

[%collapsible]

== Module summary

You deployed the DataSphere application three different ways and learned the building blocks
behind every OpenShift application.

*The three building blocks:*

* *Pod*: the running container; ephemeral and managed by a Deployment
* *Service*: stable internal network address; uses label selectors to find pods
* *Route*: external access with a public hostname and automatic TLS termination

*The three deployment methods:*

* *Container image*: fastest path when the image is already built and published;
  no build step, no source code required
* *Source-to-Image*: point OpenShift at a Git repository and it builds, packages,
  and deploys the application; no Dockerfile, no local tooling required
* *Helm chart*: package the entire multi-component application for repeatable,
  version-controlled deployments; install or remove the whole stack in one command

*Next steps:*

DataSphere is running. Lab 2 covers what happens when things break -- and how OpenShift
keeps your application available automatically with scaling and self-healing.
