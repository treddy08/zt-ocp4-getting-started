= Module 2: 3 Ways to Deploy an Application
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application running on OpenShift.
In this module you will deploy *DataSphere* — a 2-tier operations dashboard — using 3 different methods:
from a pre-built container image, from source code using Source-to-Image, and as a packaged Helm chart.

DataSphere has 2 components: a React frontend served by Nginx (`datasphere-ui`), and a Python REST API
(`datasphere-api`). The frontend talks to the API entirely by Service name — OpenShift's internal DNS
resolves it automatically.

[mermaid]
....
flowchart TB
    B["🌐 Browser"]
    R["Route\n(datasphere-ui)"]
    UI["datasphere-ui\nReact + Nginx"]
    API["datasphere-api\nPython FastAPI"]

    B -->|"HTTPS"| R
    R --> UI
    UI -->|"/api/* forwarded to\ndatasphere-api:8080"| API
....

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Deploy an application from a pre-built container image using `oc new-app`
* Trigger a Source-to-Image build from a Git repository
* Deploy a multi-component application in a single command using a Helm chart
* Understand when each deployment method is the right choice

=== Step 2.1: Deploy the UI from a container image

The UI image is already built and published to a container registry.
Deploy it with a single command:

[source,role="execute"]
----
oc new-app quay.io/rhdp/datasphere-ui:1.0.0 --name=datasphere-ui
----

[%collapsible]
.Expected output
====
----
--> Found container image ... (quay.io/rhdp/datasphere-ui:1.0.0)
    ...
--> Creating resources ...
    imagestream.image.openshift.io "datasphere-ui" created
    deployment.apps "datasphere-ui" created
    service "datasphere-ui" created
--> Success
    Application is not exposed. You can expose services to the outside world by executing one or more of the commands below:
     'oc expose service/datasphere-ui'
    Run 'oc status' to view your app.
----
====

Notice what `oc new-app` did in one shot: it created a Deployment to run and manage the container,
a Service so other components can reach the UI internally, and an ImageStream to track the image version.
No YAML written. No pod spec authored.

The output also tells you the app isn't exposed yet — there's a Service for internal access,
but nothing public-facing. Fix that now:

[source,role="execute"]
----
oc expose service datasphere-ui
----

[%collapsible]
.Expected output
====
----
route.route.openshift.io/datasphere-ui exposed
----
====

A Route was just created with a public hostname and TLS termination handled by OpenShift automatically.
Wait for the pod to come up:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
----
====

Open the URL in your browser:

[source,role="execute"]
----
oc get route datasphere-ui -o jsonpath='{.spec.host}'
----

The page loads — but the map is empty. That's not a bug. The UI is working exactly as it should:
it's already trying to reach the API at `datasphere-api:8080`, but there's no pod answering
that address yet. You'll add the API next, and the UI will find it on its own.

=== Step 2.2: Build the API from source code

The API source code lives in a Git repository.
Rather than building a container image locally and pushing it yourself, you point OpenShift at
the repository and let it do the work. This is *Source-to-Image* (S2I).

The `python~` prefix tells OpenShift two things: use the Python builder image, and the source
is what follows. OpenShift clones the repo, detects the `requirements.txt`, installs dependencies,
and packages everything into a container image — all inside the cluster, with no Dockerfile required.

[source,role="execute",subs="attributes+"]
----
oc new-app python~{datasphere_api_repo} --name=datasphere-api
----

[%collapsible]
.Expected output
====
----
--> Found image ... (Python 3.11 on UBI 9)
    ...
--> Creating resources ...
    imagestream.image.openshift.io "python" created
    imagestream.image.openshift.io "datasphere-api" created
    buildconfig.build.openshift.io "datasphere-api" created
    deployment.apps "datasphere-api" created
    service "datasphere-api" created
--> Success
    Build scheduled, use 'oc logs -f buildconfig/datasphere-api' to track its progress.
----
====

This time `oc new-app` created a `buildconfig` in addition to the Deployment and Service.
The BuildConfig is the S2I build definition — it's what OpenShift will use every time you want
to rebuild the API from source. Watch the first build run:

[source,role="execute"]
----
oc logs -f buildconfig/datasphere-api
----

[%collapsible]
.Expected output (truncated)
====
----
Cloning "{datasphere_api_repo}" ...
STEP 1/9: FROM image-registry.openshift-image-registry.svc:5000/openshift/python:latest
...
Collecting fastapi
Collecting uvicorn
...
Successfully installed fastapi-0.115.0 uvicorn-0.31.0 ...
Pushing image image-registry.openshift-image-registry.svc:5000/{user}-lab/datasphere-api:latest ...
Push successful
----
====

That entire build ran inside the cluster: no Docker installed locally, no CI/CD pipeline configured,
no image registry account needed. OpenShift pulled the code, installed dependencies, built and
tagged the image, pushed it to its internal registry, and is now deploying it — all from
one `oc new-app` command.

Wait for the deployment to finish:

[source,role="execute"]
----
oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-api" successfully rolled out
----
====

=== Step 2.3: Verify

Open the UI URL again:

[source,role="execute"]
----
oc get route datasphere-ui -o jsonpath='{.spec.host}'
----

The map is now populated with datacenter sites — and you didn't touch the UI at all.
The moment `datasphere-api` came online, the UI found it automatically. OpenShift's internal DNS
resolved the Service name `datasphere-api` to the running pod, and the Nginx reverse proxy
started forwarding requests to it.

This is service discovery in practice: components find each other by name, not by IP address.
Add a backend, and the frontend connects to it. No reconfiguration required.

== Section 3: Deploy the full stack as a Helm chart (3 minutes)

You just deployed DataSphere in a series of individual commands — each one creating a specific
resource. Helm packages all of those resources into a single installable unit called a *chart*.

A Helm chart for DataSphere defines the same Deployments, Services, and Route you created manually,
but with one important difference: the entire application installs, upgrades, and removes as
a single operation. This is how teams package applications for repeatable deployments.

Create a new Project:

[source,role="execute",subs="attributes+"]
----
oc new-project {user}-datasphere-helm
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
Now using project "{user}-datasphere-helm" on server "{openshift_api_url}".
----
====

The DataSphere Helm chart is pre-staged in your home directory. Install it:

[source,role="execute"]
----
helm install datasphere ~/datasphere-gitops
----

[%collapsible]
.Expected output
====
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-datasphere-helm
STATUS: deployed
REVISION: 1
----
====

Wait for both components:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui && oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
deployment "datasphere-api" successfully rolled out
----
====

Open the application:

[source,role="execute"]
----
oc get route datasphere-ui -o jsonpath='{.spec.host}'
----

The same 2-tier application — fully operational. Every resource you created manually in Section 2
was created here by a single command. That's what Helm gives you: the same result, with no
opportunity for steps to be missed or applied in the wrong order.

NOTE: To remove the entire application: `helm uninstall datasphere`

== Module summary

You deployed the same 2-tier application using 3 different methods.

*The 3 deployment methods:*

* *Container image*: fastest path when the image is already built and published. No build step, no source code required.
* *Source-to-Image*: point OpenShift at a Git repository and it builds, packages, and deploys the application. No Dockerfile, no local tooling required.
* *Helm chart*: package the entire multi-component application for repeatable, version-controlled deployments. Install or remove the whole stack in one command.
