= Module 2: Expanding and Automating Your Application Deployment (Web Console)
:source-highlighter: rouge
:toc: macro
:toclevels: 2

In this module you will add the DataSphere backend API to the frontend you deployed in Lab 1,
using *Source-to-Image* — OpenShift builds and deploys directly from your Git repository,
no Dockerfile required. Then you'll deploy the complete 2-tier application as a *Helm chart*.

If you completed Lab 1, your `datasphere-ui` is already running in `{user}-lab` and the map is
empty. This lab adds the API and fills it in.

If you're starting here, the <<quick-start>> section gets you to the same point in 2 minutes.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Explain what Source-to-Image (S2I) does and when to use it
* Trigger an S2I build from a Git repository using the web console
* Explain how OpenShift's internal DNS connects components by Service name
* Deploy a multi-component application in a single command using a Helm chart
* Understand when to reach for each of the 3 deployment methods

== Section 2: Explore the architecture (5 minutes)

=== The frontend: datasphere-ui

In the Topology view, click the `datasphere-ui` component to open its side panel.
Click *Actions* → *Edit Deployment* to see its configuration.

Under *Container*, note the image: `{datasphere_ui_image}`.

No source code was compiled. No build was needed.

image::console-guide/lab-02-03-datasphere-ui-image.png[datasphere-ui Deployment edit view showing the container image field]

=== The backend: datasphere-api

Click the `datasphere-api` component in the Topology view.

The API image was built from source code using Source-to-Image at provisioning time and pushed
to the cluster's internal registry.

image::console-guide/lab-02-05-datasphere-api-side-panel.png[datasphere-api side panel showing the component with no Routes]

=== How they connect

In the left navigation, click *Project* → *Services*. Notice that `datasphere-api` has
no Route: it's only reachable from within the cluster.
The UI talks to it by Service name. OpenShift's internal DNS resolves that name to the correct
pod automatically, even as pods restart and change IP addresses.

image::console-guide/lab-02-06-services-list.png[Services list showing both datasphere-ui and datasphere-api Services — no Route for datasphere-api]

== Section 3: Add the backend via Source-to-Image (8 minutes)

Switch to your lab Project by clicking the *Project* dropdown and selecting `{user}-lab`.

`datasphere-ui` is already running here from Lab 1. The map is empty because there's no API
yet — the frontend is already calling `datasphere-api:8080` on every page load, but nothing is
answering. That changes now.

=== Step 3.1: Build the API from source code

Click *+Add* in the left navigation and select *Import from Git*.

image::console-guide/lab-02-04-add-menu.png[Developer perspective +Add menu showing deployment options]

image::console-guide/lab-02-09-import-from-git-form.png[Import from Git form showing the Git Repo URL field]

Fill in the form:

* *Git Repo URL*: `{gitea_url}/{user}/datasphere-api`
* Wait for the repository to be validated
* OpenShift auto-detects the Python language and selects the *Python* builder image

NOTE: This is Source-to-Image (S2I) in action. OpenShift reads the repository, identifies the
language, and selects the right builder automatically. No Dockerfile required.

Scroll down and set:

* *Application*: select `datasphere` (same application group as the UI)
* *Name*: `datasphere-api`
* Under *Advanced options*, *uncheck* *Create a route to the Application*

The API only needs to be reachable from within the cluster — it doesn't need a public URL.

image::console-guide/lab-02-10-import-git-python-builder.png[Import from Git form showing Python builder auto-selected, datasphere-api name, and no Route option]

Click *Create*.

In the Topology view, the `datasphere-api` component appears with a build animation.
Click on the component and select the *Builds* tab to watch the build progress.

DataSphere UI and reload the page.

The map is now populated with datacenter sites. The UI found the API automatically — no

== Section 4: Deploy the full stack as a Helm chart (5 minutes)

You deployed DataSphere manually in 2 steps using the console.
Helm lets you package all of those resources into a single installable unit.

This section uses the terminal. Click the *Terminal* tab on the right side of this lab interface.

Switch to your pre-provisioned Helm Project and install the DataSphere chart:

[source,role="execute",subs="attributes+"]
----
oc project {user}-lab-helm && helm install datasphere {gitea_url}/api/packages/gitea-admin/helm/datasphere-{datasphere_helm_version}.tgz
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-lab-helm
STATUS: deployed
REVISION: 1
----
====

Wait for both pods to come up:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui && oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
deployment "datasphere-api" successfully rolled out
----
====

Switch back to the *Console* tab in the Showroom interface.
In the web console, switch to the `{user}-lab-helm` Project and open the Topology view.

//TODO: add screenshot of Topology for the Helm-deployed project

Both components appear — identical to what you built manually, deployed from a single command.

Click the *open URL* icon on `datasphere-ui` to confirm the application is working.

When you're done, remove the entire application from the terminal:

[source,role="execute"]
----
helm uninstall datasphere
----

[%collapsible]
.Expected output
====
----
release "datasphere" uninstalled
----
====

Watch the Topology view in the console — both components disappear as the resources are deleted.

== Module summary

You completed the DataSphere deployment using 3 different methods across Lab 1 and Lab 2.

*What you accomplished:*

* Built the DataSphere API from a Git repository using *Import from Git* (Source-to-Image)
* Watched OpenShift build, package, and deploy the API — no Dockerfile required
* Observed how OpenShift's internal DNS connected the UI to the API automatically by Service name
* Deployed the complete 2-tier stack with a single `helm install` command

*The 3 deployment methods:*

* *Container image* (Lab 1): fastest path when the image is already built and published
* *Source-to-Image* (this lab): point OpenShift at a Git repository and it builds, packages, and deploys; no Dockerfile required
* *Helm chart* (this lab): package the entire multi-component application for repeatable, version-controlled deployments
