= Module 2: 3 Ways to Deploy an Application (Web Console)
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application running on OpenShift.
In this module you will deploy *DataSphere* — a 2-tier operations dashboard — using 3 different methods:
from a pre-built container image, from source code using Source-to-Image, and as a packaged Helm chart.

Each method has its place, and by the end you will know when to reach for each one.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Deploy an application from a pre-built container image using the web console
* Trigger a Source-to-Image build from a Git repository using the web console
* Deploy a multi-component application in a single command using a Helm chart
* Understand when each deployment method is the right choice

to see the connection details.

In the left navigation, click *Project* → *Services*. Notice that `datasphere-api` has

The UI talks to it by Service name. OpenShift's internal DNS resolves that name to the correct
pod automatically, even as pods restart and change IP addresses.

Click *+Add* in the left navigation (or the prompt in the Topology canvas).

=== Step 3.2: Build the API from source code

Click *+Add* and select *Import from Git*.

//TODO: add screenshot of Import from Git form

Fill in the form:

* *Git Repo URL*: `{datasphere_api_repo}`
* Wait for the repository to be validated
* OpenShift auto-detects the Python language and selects the *Python* builder image

NOTE: This is Source-to-Image (S2I) in action. OpenShift reads the repository, identifies the
language, and selects the right builder automatically. No Dockerfile required.

Scroll down and set:

* *Application*: select `datasphere` (same application group as the UI)
* *Name*: `datasphere-api`
* Under *Advanced options*, *uncheck* *Create a route to the Application*

The API only needs to be reachable from within the cluster — it doesn't need a public URL.

== Section 4: Deploy the full stack as a Helm chart (5 minutes)

You deployed DataSphere manually in 2 steps using the console.
Helm lets you package all of those resources into a single installable unit.

This section uses the terminal. Click the *Terminal* tab on the right side of this lab interface.

Create a new Project:

[source,role="execute",subs="attributes+"]
----
oc new-project {user}-datasphere-helm
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
Now using project "{user}-datasphere-helm" on server "{openshift_api_url}".
----
====

The DataSphere Helm chart is pre-staged in your home directory. Install it:

[source,role="execute"]
----
helm install datasphere ~/datasphere-gitops
----

[%collapsible]
.Expected output
====
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-datasphere-helm
STATUS: deployed
REVISION: 1
NOTES:
DataSphere deployed. Get the UI URL with:
  oc get route datasphere-ui -o jsonpath='{.spec.host}'
----
====

Wait for both pods to come up:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui && oc rollout status deployment/datasphere-api
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
deployment "datasphere-api" successfully rolled out
----
====

Switch back to the *Console* tab in the Showroom interface.
In the web console, switch to the `{user}-datasphere-helm` Project and open the Topology view.

//TODO: add screenshot of Topology for the Helm-deployed project

Both components appear — identical to what you built manually, deployed from a single command.

Click the *open URL* icon on `datasphere-ui` to confirm the application is working.

When you're done, remove the entire application from the terminal:

[source,role="execute"]
----
helm uninstall datasphere
----

[%collapsible]
.Expected output
====
----
release "datasphere" uninstalled
----
====

Watch the Topology view in the console — both components disappear as the resources are deleted.

== Module summary

You deployed the same 2-tier application using 3 different methods.

*What you accomplished:*

* Deployed a pre-built container image using *Container images* in the web console
* Built a Python application from a Git repository using *Import from Git* (Source-to-Image)
* Deployed the complete stack with `helm install`
* Observed how OpenShift's internal DNS connects components automatically by Service name

*The 3 deployment methods:*

* *Container image*: fastest path when the image is already built and published
* *Source-to-Image*: build directly from a Git repository; OpenShift handles the container build
* *Helm chart*: package the entire application for repeatable, version-controlled deployments
