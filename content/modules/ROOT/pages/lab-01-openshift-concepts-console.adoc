= Module 1: Your First Application (Web Console)
:source-highlighter: rouge
:toc: macro
:toclevels: 2

In this module, you'll deploy the DataSphere frontend on OpenShift and learn the 3 building
blocks that make every OpenShift application work: *Pods*, *Services*, and *Routes*.

First you'll see the finished result — the full DataSphere application already running, live map
and all. Then you'll explore what OpenShift created to make it work. Finally, you'll deploy
the frontend yourself from scratch using the web console.

By the end, the map will be empty. That's intentional. Lab 2 picks up here.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application using the OpenShift web console
* Expose a deployment internally with a Service and externally with a Route
* Verify application health using the web console

== Section 2: Explore the pieces (5 minutes)

Every application on OpenShift is built from the same 3 core resources.
Let's inspect each one using the `datasphere-ui` component.

=== The Pod: where your application runs

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives them
a shared network and storage context.

. In the left navigation, click *Workloads* -> *Pods*.
+
You will see 2 pods listed, both with *Running* status and *1/1* in the Ready column.

image::console-guide/lab-01-04-pods-list.png[Pods list showing both datasphere-ui and datasphere-api pods Running with 1/1 Ready]

. Click the `datasphere-ui-` pod (the one with the longer random suffix) to view its details.
+
The Pod details page shows:
+
* *Name*: the unique pod identifier (includes a random suffix)
* *Namespace*: `{user}-datasphere`
* *Labels*: `app=datasphere,component=ui` — the labels that connect this pod to its Service
* *Status*: Running
* *Pod IP*: the internal cluster IP assigned to this pod
* *Container image*: `{datasphere_ui_image}`

image::console-guide/lab-01-05-pod-details.png[Pod details page showing status Running, labels app=datasphere component=ui, and container image]

. Click the *Logs* tab to check the application logs.
+
You will see nginx access log entries, confirming the web server is running inside the container.

image::console-guide/lab-01-06-pod-logs.png[Pod Logs tab showing the container log viewer]

NOTE: Pods are ephemeral. They can be created, destroyed, and replaced at any time.
A *Deployment* manages this lifecycle and ensures the right number of pods are always running.

=== The Service: internal network access

A *Service* gives your application a stable internal network address. Even if the Pod restarts and
gets a new IP address, the Service address stays the same.

. In the left navigation, click *Networking* -> *Services*, then click *datasphere-ui*.
+
The Service details page shows how internal networking works:
+
* *Pod selector*: `app=datasphere,component=ui` — this label determines which pods receive traffic from this Service
* *Service address*: A *Cluster IP* (accessible within the cluster only)
* *Hostname*: The internal DNS name other applications can use to reach this service
* *Service port mapping*: Port *8080/TCP* — matching the container's listening port

image::console-guide/lab-01-07-service-details.png[Service details for datasphere-ui showing Pod selector, Cluster IP, and port 8080 mapping]

Notice the `Pod selector` field: it uses *labels* to identify which pods the Service should route
traffic to. This label-based selection is how OpenShift connects resources together.

=== The Route: external access

A *Route* exposes a Service to traffic from outside the cluster. It provides a public hostname and
handles TLS termination so your users can access the app over HTTPS.

. Navigate to *Networking* -> *Routes* and click on *datasphere-ui*.
+
The Route details page shows how external access works:
+
* *Location*: The public URL users can access from anywhere on the internet
* *Host*: The hostname that OpenShift generated automatically
* *Service*: Links to the *datasphere-ui* Service, which the Route forwards traffic to
* *Target port*: `8080`
* *TLS*: Edge termination with redirect — OpenShift handles the certificate automatically
* *Status*: *Accepted* — the Route is active and serving traffic

image::console-guide/lab-01-08-route-details.png[Route details showing public URL, datasphere-ui service link, and TLS edge termination]

=== Why datasphere-api has no Route

Go back to *Networking* -> *Routes*. There is only 1 Route — for `datasphere-ui`.
`datasphere-api` has no Route. That's deliberate: the API only needs to be reachable from within
the cluster, not from the internet.

Navigate to *Networking* -> *Services*. You'll see Services for both `datasphere-ui` and
`datasphere-api`. The UI talks to the API by Service name — `datasphere-api` — and OpenShift's
internal DNS resolves it automatically. No Route needed.

=== How they connect

[mermaid]
....
flowchart TB
    B["🌐 Browser"]
    R["Route\n(datasphere-ui)"]
    S["Service\n(stable internal IP)"]
    P["Pod\n(running container)"]

    B -->|"HTTPS"| R
    R --> S
    S -->|"label selector"| P
....

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to via label selectors.
The Pod runs your container.

== Section 3: Build it yourself (8 minutes)

Now that you understand the pieces, let's build the frontend from scratch.

=== Switch to your clean Project

. Click the *Project* dropdown at the top of the page and select *{user}-lab*.
. Navigate to *Workloads* -> *Deployments* to verify the project is empty. You should see *No Deployments found*.

image::console-guide/lab-01-09-empty-project.png[Empty Deployments page in the clean lab project showing No Deployments found]

=== Step 3.1: Deploy the application

. On the Deployments page, click the *Create Deployment* button in the top right.

. In the Create Deployment form:
+
.. Enter `datasphere-ui` in the *Name* field
+
image::console-guide/lab-01-10-create-deployment-name.png[Create Deployment form with datasphere-ui entered in the Name field]
+
.. Scroll down to the *Images* section
.. Uncheck *Deploy image from an image stream tag*
.. In the *Image Name* field, enter: `{datasphere_ui_image}`
+
image::console-guide/lab-01-11-create-deployment-image.png[Images section with Deploy image from an image stream tag unchecked and container image name entered]
+
.. Click *Create*

You will be taken to the Deployment details page for `datasphere-ui`. The pod ring shows your pod starting up.

image::console-guide/lab-01-12-deployment-created.png[Deployment details page showing datasphere-ui created with pod ring starting up]

This creates a *Deployment* resource. OpenShift uses the Deployment to create and manage your Pod:
if the Pod crashes, the Deployment automatically restarts it.

==== Verify

Navigate to *Workloads* -> *Pods* to verify your pod is running.

You should see your `datasphere-ui` pod listed with:

* *Status*: Running
* *Ready*: 1/1

image::console-guide/lab-01-13-pods-running.png[Pods page showing datasphere-ui pod with Running status and 1/1 Ready]

=== Step 3.2: Verify the Service

`oc new-app` creates a Service automatically. In the console, "Create Deployment" does not —
you need to create one explicitly. Let's do that now using the YAML editor.

. Click the *+* icon in the top navigation bar to open *Import YAML*.

. Replace any existing content with the following Service definition:
+
[source,yaml,subs="attributes+"]
----
apiVersion: v1
kind: Service
metadata:
  name: datasphere-ui
  namespace: {user}-lab
spec:
  selector:
    app: datasphere-ui
  ports:
    - protocol: TCP
      port: 8080
      targetPort: 8080
----

. Click *Create*.

The Service details page shows your new `datasphere-ui` Service with:

* *Pod selector*: `app=datasphere-ui` — this label matches the pods created by your Deployment
* *Type*: ClusterIP — a stable internal address
* *Service port mapping*: port 8080/TCP forwarding to pod port 8080

image::console-guide/lab-01-14-service-details.png[Service details for datasphere-ui showing Pod selector, ClusterIP, and port 8080 mapping]

The `ClusterIP` is the stable internal address. Other applications inside the cluster can reach
your app at this address, but it's not yet accessible from outside.

=== Step 3.3: Create a Route for external access

. In the left navigation, click *Networking* -> *Routes*.
+
You'll see an empty list since no Routes exist yet.
+
image::console-guide/lab-01-15-routes-empty.png[Routes page showing no routes — empty list before creating the Route]

. Click the *Create Route* button.
+
The console opens a YAML editor pre-populated with a Route template.

. Replace the contents with the following Route definition:
+
[source,yaml,subs="attributes+"]
----
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: datasphere-ui
  namespace: {user}-lab
spec:
  to:
    kind: Service
    name: datasphere-ui
  port:
    targetPort: 8080
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
----
+
image::console-guide/lab-01-16-create-route-yaml.png[YAML editor showing the datasphere-ui Route definition]

. Click *Create*.

The Route details page confirms your Route is active:

* *Status*: Accepted
* *Location*: a public URL for your application
* *Service*: datasphere-ui

image::console-guide/lab-01-17-route-details.png[Route details page showing datasphere-ui Route with public URL, Accepted status, and TLS edge termination]

==== Verify

Click the *Location* link on the Route details page to open the application URL in your browser.

The DataSphere UI loads — but the map is empty. No datacenter sites, no status data.

That's not a bug. The UI is running exactly as it should: it's already calling out to
`datasphere-api:8080`, looking for the backend. But there's no pod answering that address yet.
You've deployed the frontend. The backend doesn't exist in this Project.

In Lab 2, you'll add the API using *Source-to-Image* — OpenShift will build it directly from
source code — and the moment it comes online, the map will populate automatically. No changes
to the UI required.

image::console-guide/lab-01-18-datasphere-empty-map.png[DataSphere UI loaded in the browser showing an empty map with no datacenter sites]

== Module summary

You've deployed the frontend tier of a real 2-tier application on OpenShift using the web console.

*What you accomplished:*

* Explored a pre-deployed application and identified its Pods, Services, and Routes
* Understood how the 3 core resources connect to serve external traffic
* Understood why some components have Routes and others don't
* Deployed a container image to OpenShift using the Create Deployment form
* Created a Service to give your app a stable internal network address
* Created a Route to expose your app to the internet with automatic HTTPS

*Key concepts:*

* *Pod*: the running container; ephemeral and managed by a Deployment
* *Service*: stable internal network address; uses label selectors to find pods
* *Route*: external access with a public hostname and automatic TLS termination
* *Labels*: key/value metadata that connects resources together

*Next steps:*

The frontend is running but the map is empty. Lab 2 adds the backend using Source-to-Image,
then shows how to deploy the complete 2-tier application as a single Helm chart.
If you already know Pods, Services, and Routes, you can start Lab 2 directly.
