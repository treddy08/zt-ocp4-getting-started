= Module 2: Expanding and Automating Your Application Deployment
:source-highlighter: rouge
:toc: macro
:toclevels: 2

There is no single right way to get an application running on OpenShift.
In this module you will deploy the DataSphere backend API using Source-to-Image — OpenShift
builds and deploys directly from your Git repository, no Dockerfile required. Then you'll
deploy the complete 2-tier application as a Helm chart.

If you completed Lab 1, your `datasphere-ui` is already running in `{user}-lab` and the map is
empty. This lab adds the API and fills it in.

If you're starting here, the <<quick-start>> section gets you to the same point in 2 minutes.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Explain what Source-to-Image (S2I) does and when to use it
* Trigger an S2I build from a Git repository using `oc new-app`
* Explain how OpenShift's internal DNS connects components by Service name
* Deploy a multi-component application in a single command using a Helm chart
* Understand when to reach for each of the 3 deployment methods

NOTE: This module does not re-explain Pods, Services, or Routes. If those are new concepts,
start with Lab 1 first.

== Quick start (skip if you completed Lab 1)

If you're starting here, run these commands to deploy `datasphere-ui` and reach the same
starting point as Lab 1 graduates.

Log in to the cluster. To open a terminal, click the *Terminal* tab on the right side of
this lab interface:

[source,role="execute",subs="attributes+"]
----
oc login {openshift_api_url} --username {user} --password {password}
----

Switch to your lab Project, deploy the DataSphere UI, and expose it:

[source,role="execute",subs="attributes+"]
----
oc project {user}-lab
oc new-app {datasphere_ui_image} --name=datasphere-ui
oc create route edge --service=datasphere-ui --insecure-policy=Redirect
oc rollout status deployment/datasphere-ui
----

Get the URL and open it in your browser — the map will be empty. That's expected:

[source,role="execute"]
----
oc get route datasphere-ui
----

Copy the URL from the `HOST` column. Continue to <<section-1>>.

=== Step 2.1: Find your Gitea repository

Your lab environment includes a personal Gitea instance with the DataSphere API source already
loaded. Open Gitea from the URL in your connection details and log in with your lab credentials
(`{user}` / `{password}`). You should see a `datasphere-api` repository in your account —
that's the source code you'll build from in the next step.

=== Step 2.2: Build and deploy the API from source

Rather than building a container image locally and pushing it yourself, you point OpenShift at
your Gitea repository and let it do the work. This is *Source-to-Image* (S2I).

The `python~` prefix tells OpenShift two things: use the Python builder image, and build from
the source that follows. OpenShift clones the repository, detects `requirements.txt`,
installs dependencies, and packages everything into a container image — all inside the cluster,
with no Dockerfile required.

[source,role="execute",subs="attributes+"]
----
oc new-app python:3.12-ubi9~{gitea_url}/{user}/datasphere-api --name=datasphere-api
----

[%collapsible]
.Expected output
====
----
--> Found image ... (Python 3.12 on UBI 9)
    ...
--> Creating resources ...
    imagestream.image.openshift.io "python" created
    imagestream.image.openshift.io "datasphere-api" created
    buildconfig.build.openshift.io "datasphere-api" created
    deployment.apps "datasphere-api" created
    service "datasphere-api" created
--> Success
    Build scheduled, use 'oc logs -f buildconfig/datasphere-api' to track its progress.
----
====

Notice the `buildconfig` in the output — that wasn't there when you deployed the UI from a
pre-built image. The BuildConfig is the S2I build definition. OpenShift is now cloning the
repository and running the build inside the cluster. Watch it happen:

[source,role="execute"]
----
oc logs -f buildconfig/datasphere-api
----

[%collapsible]
.Expected output (truncated)
====
----
Cloning "https://gitea.../datasphere-api" ...
STEP 1/9: FROM image-registry.openshift-image-registry.svc:5000/openshift/python:latest
...
Collecting fastapi
Collecting uvicorn
...
Successfully installed fastapi-0.115.0 uvicorn-0.31.0 ...
Pushing image image-registry.openshift-image-registry.svc:5000/{user}-lab/datasphere-api:latest ...
Push successful
----
====

That entire build ran inside the cluster: no Docker installed locally, no CI/CD pipeline
configured, no external registry account needed. OpenShift pulled the source from your Gitea
repository, installed dependencies, built and tagged the image, pushed it to its internal
registry, and is now deploying it — all triggered by one `oc new-app` command.

Wait for the deployment:

[source,role="execute"]
----

----

[%collapsible]

====

----
====

Open the URL from the `HOST` column. This time the map is populated with datacenter sites —
instead of the empty map from before, you're seeing live data from the API you just deployed. You didn't reconfigure the UI. You didn't update

Project, the UI found it automatically.

This is OpenShift's internal DNS in practice: the Nginx proxy in the UI container is configured
to forward `/api/` requests to `datasphere-api:8080`. OpenShift resolves that name to the
Service, which routes to the pod. Add a backend, the frontend connects to it. That's it.

== Section 3: Deploy the full stack as a Helm chart (3 minutes)

You deployed DataSphere in 2 stages: the UI in Lab 1 (or Quick Start), the API in Section 2.
Helm packages all of those resources into a single installable unit — a *chart*.

A chart is a directory of templated Kubernetes manifests plus a `values.yaml` file that controls
configurable inputs (image tags, theme, replica count, and so on). When you run `helm install`,
Helm renders those templates with your chosen values and submits all the resulting manifests to
the cluster in one shot. It also records what it installed as a named *release*, so it can
upgrade, roll back, or cleanly remove every resource it owns.

The DataSphere chart defines the same Deployments, Services, and Route you created manually
across the last two sections — but as a single, versioned, repeatable unit. Every team member
who runs this command gets the same result.

Switch to your pre-provisioned Helm Project:

[source,role="execute",subs="attributes+"]
----
oc project {user}-lab-helm
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
Now using project "{user}-lab-helm" on server "{openshift_api_url}".
----
====

Install the DataSphere chart:

[source,role="execute",subs="attributes+"]
----
helm install datasphere {gitea_url}/api/packages/gitea-admin/helm/datasphere-{datasphere_helm_version}.tgz
----

The first argument after `helm install` is the *release name* (`datasphere`). Helm uses it to
track this installation. The second argument is the chart URL — it points directly at the
DataSphere chart package hosted in your lab's Gitea instance.

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
NAME: datasphere
LAST DEPLOYED: ...
NAMESPACE: {user}-lab-helm
STATUS: deployed
REVISION: 1
----
====

`REVISION: 1` tells you this is the first install. Each `helm upgrade` increments the revision
and Helm keeps the history — you can roll back to any previous revision if something goes wrong.

Wait for both components to roll out:

[source,role="execute"]
----

----

[%collapsible]
.Expected output
====
----

----
====

[source,role="execute"]

----

[%collapsible]
.Expected output

----
====

Open the application:

[source,role="execute"]

Both tiers, fully operational, from a single command. Every resource you created across Lab 1
and Lab 2 was created here in one shot — with no opportunity for steps to be missed or
applied in the wrong order.

== Module summary

You completed the DataSphere deployment using 3 different methods.

*The 3 deployment methods:*

* *Container image* (Lab 1): fastest path when the image is already built and published;
  no build step, no source code required
* *Source-to-Image* (this lab): point OpenShift at a Git repository and it builds, packages,
  and deploys the application; no Dockerfile, no local tooling required
* *Helm chart* (this lab): package the entire multi-component application for repeatable,
  version-controlled deployments; install or remove the whole stack in one command
