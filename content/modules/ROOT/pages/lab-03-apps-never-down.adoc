= Module 3: Apps That Never Go Down
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Your DataSphere demo app (the datacenter visualization you deployed in Labs 1 and 2) is
running. Users are happy. But what happens when a pod crashes? What happens when traffic
spikes at 3am? In this module, you'll find out by breaking things on purpose, then
watching OpenShift fix them automatically.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Explain how ReplicaSets maintain pod availability
* Configure a Horizontal Pod Autoscaler (HPA) for automatic scaling
* Describe the difference between liveness and readiness probes
* Demonstrate zero-downtime resilience under pod failure and traffic load

== Section 1: What happens when things break?

The DataSphere demo app is already deployed with resilience features configured: health
probes, resource requests, and an auto-scaler. In this section, you'll interact with
those features by breaking things and watching OpenShift respond.

=== Kill a pod

Check how many API pods are running:

[source,role="execute",subs="attributes+"]
----
oc get pods --selector component=api
----

[%collapsible]
.Expected output
====
----
NAME                              READY   STATUS    RESTARTS   AGE
datasphere-api-5d4f8b7c9-abc12   1/1     Running   0          10m
----
====

1 pod. Delete it:

[source,role="execute",subs="attributes+"]
----
oc delete pod --selector component=api
----

[%collapsible]
.Expected output
====
----
pod "datasphere-api-5d4f8b7c9-abc12" deleted
----
====

Switch to the browser. Watch for the "API unreachable" warning in the top status bar.
The warning appears briefly, then clears as the replacement pod starts.

Check the pods again:

[source,role="execute",subs="attributes+"]
----
oc get pods --selector component=api
----

[%collapsible]
.Expected output
====
----
NAME                              READY   STATUS    RESTARTS   AGE
datasphere-api-5d4f8b7c9-xyz99   1/1     Running   0          8s
----
====

OpenShift replaced the pod automatically through the ReplicaSet controller. But your
users experienced a real outage during the gap. With only 1 replica, any pod failure
is user-visible.

=== Fix it with replicas

Scale the API deployment to 3 replicas:

[source,role="execute",subs="attributes+"]
----
oc scale deployment datasphere-api --replicas=3
----

[%collapsible]
.Expected output
====
----
deployment.apps/datasphere-api scaled
----
====

Watch the new pods come up:

[source,role="execute",subs="attributes+"]
----
oc get pods --selector component=api
----

[%collapsible]
.Expected output
====
----
NAME                              READY   STATUS    RESTARTS   AGE
datasphere-api-5d4f8b7c9-abc12   1/1     Running   0          5m
datasphere-api-5d4f8b7c9-def34   1/1     Running   0          12s
datasphere-api-5d4f8b7c9-ghi56   1/1     Running   0          12s
----
====

Now delete a pod again:

[source,role="execute",subs="attributes+"]
----
oc delete pod --selector component=api --field-selector=status.phase=Running --wait=false
----

[NOTE]
====
The `--wait=false` flag returns immediately so you can check the app right away.
This command deletes all running API pods, but the ReplicaSet will create 3
replacements.
====

Switch to the browser. This time: no warning. The map stayed live the entire time.
The Service routed traffic to the surviving pods while replacements started.

[NOTE]
====
With multiple replicas, the Service load-balances across all pods with matching labels.
Losing 1 pod out of 3 means the other 2 handle requests without interruption. This is
why production workloads always run more than 1 replica.
====

=== Auto-scaling under load

You just scaled manually. That works, but it requires you to be watching. What if
traffic spikes at 3am? The Horizontal Pod Autoscaler (HPA) handles this automatically.

First, check the current HPA status:

[source,role="execute",subs="attributes+"]
----
oc get hpa
----

[%collapsible]
.Expected output
====
----
NAME              REFERENCE                    TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
datasphere-api    Deployment/datasphere-api    2%/50%    1         5         3          15m
----
====

The `TARGETS` column shows `2%/50%`: current CPU utilization is 2%, and the HPA will
scale up when it crosses 50%.

Now look at the DataSphere app in your browser. At the bottom of the page, you'll see
the *Load Control Panel*: a slider, plus real-time stats showing simulated traffic
percentage, per-pod load, and the current replica count.

In production, CPU load increases because more users arrive or workloads get heavier.
The DataSphere demo app simulates this: the slider creates CPU-intensive computation
on every API request, like a real app processing heavier queries under load. The
effect is the same: each pod's CPU usage climbs, and eventually the platform needs
to add more capacity.

Drag the slider to about *80%*.

Watch from the terminal:

[source,role="execute"]
----
watch oc get hpa
----

[%collapsible]
.Expected output (after ~30 seconds)
====
----
NAME              REFERENCE                    TARGETS    MINPODS   MAXPODS   REPLICAS   AGE
datasphere-api    Deployment/datasphere-api    78%/50%    1         5         5          16m
----
====

The CPU percentage climbs past the 50% threshold, and the HPA increases the replica
count from 3 to 4, then 5.

Switch to the browser. The Load Control Panel shows the per-pod load dropping as new
pods join and share the work.

Now slide back to *0%*.

Watch the terminal. The HPA scales the pods back down over about 20 seconds as CPU
usage drops.

Press `Ctrl+C` to stop the watch.

[NOTE]
====
The HPA scales up quickly (within 15 seconds of crossing the threshold) but scales
down more slowly. We've set the scale-down stabilization window to 20 seconds for
this demo. In production, the default is 5 minutes: this prevents the HPA from
flapping if load fluctuates up and down.
====

You just saw the platform do what you did manually in the previous step, but without
you being there. No alerts, no manual intervention, no 3am page.

=== Health probe: detect and fix a sick pod

What if a pod is still running but broken? A crashed pod gets replaced by the
ReplicaSet, but a sick pod that's still alive needs a different mechanism: the
liveness probe.

The DataSphere API has a liveness probe configured. It checks the `/health` endpoint
every 10 seconds. If the check fails 3 times in a row, OpenShift restarts the
container.

Trigger a health failure on one pod:

[source,role="execute",subs="attributes+"]
----
curl -s -X POST https://$(oc get route datasphere-ui -o jsonpath='{.spec.host}')/api/break
----

[%collapsible]
.Expected output
====
----
{"status":"broken","message":"Health check will now fail. OCP will restart this pod."}
----
====

[NOTE]
====
This request hit 1 of the API pods through the Service load balancer. Only that pod
has its health flag set to broken. The other pods remain healthy: the broken state is
stored in memory, per-pod.
====

Watch the pods:

[source,role="execute"]
----
watch oc get pods --selector component=api
----

Within 30 seconds, you'll see the RESTARTS column increment on one pod. The liveness
probe detected the failure and restarted the container automatically.

Press `Ctrl+C` to stop the watch.

Switch to the browser. The map never went down: the other replicas kept serving
requests while the broken pod was restarted.

[NOTE]
====
Notice the difference: in the first demo, deleting a pod removed it entirely and the
ReplicaSet created a replacement. Here, the liveness probe *restarted* the existing
container in place. Same result for users (no outage with multiple replicas), but a
different mechanism. Probes catch sick applications; ReplicaSets catch dead ones.
====

=== Add auto-scaling

You already scaled manually in Section 1. Now let's make it automatic.

First, set resource requests. The HPA needs CPU metrics to make scaling decisions.
Resource requests define the baseline: "25 millicores is what this pod needs at rest."
Without this, the HPA has nothing to measure against.

[source,role="execute"]
----
oc set resources deployment/datasphere-api --requests=cpu=25m,memory=64Mi --limits=cpu=100m,memory=128Mi
----

[%collapsible]
.Expected output
====
----
deployment.apps/datasphere-api resource requirements updated
----
====

Now create the HPA:

[source,role="execute"]
----
oc autoscale deployment/datasphere-api --min=1 --max=5 --cpu-percent=50
----

[%collapsible]
.Expected output
====
----
horizontalpodautoscaler.autoscaling/datasphere-api autoscaled
----
====

Check the HPA:

[source,role="execute"]
----
oc get hpa
----

[%collapsible]
.Expected output
====
----
NAME              REFERENCE                    TARGETS         MINPODS   MAXPODS   REPLICAS   AGE
datasphere-api    Deployment/datasphere-api    <unknown>/50%   1         5         1          10s
----
====

The `<unknown>` for targets is normal. It takes about 60 seconds for the metrics
server to report the first CPU measurement.

==== Explore HPA settings

Look at the full HPA specification:

[source,role="execute"]
----
oc describe hpa datasphere-api
----

[%collapsible]
.Expected output (key fields)
====
----
Name:                          datasphere-api
Reference:                     Deployment/datasphere-api
Metrics:                       ( current / target )
  resource cpu on pods:        <unknown> / 50%
Min replicas:                  1
Max replicas:                  5
Conditions:
  AbleToScale     True    ...
  ScalingActive   False   ...
  ScalingLimited  False   ...
----
====

A few settings worth knowing:

* **`--cpu-percent` threshold**: We set 50%. You could set 70% for apps that handle load
  spikes well, or 30% for latency-sensitive services that need headroom before scaling
  kicks in.

* **Scale-down stabilization window**: How long to wait before removing pods after load
  drops. The pre-deployed DataSphere uses 20 seconds for demo purposes. The production
  default is 5 minutes: this prevents the HPA from oscillating if load fluctuates up
  and down.

* **Multiple metrics**: HPA can scale on CPU, memory, custom metrics, or even external
  metrics like message queue depth. CPU is the most common starting point.

=== Add health probes

Add a liveness probe. This tells OpenShift to restart the container if the application
stops responding:

[source,role="execute"]
----
oc set probe deployment/datasphere-api --liveness --get-url=http://:8080/health --initial-delay-seconds=10 --period-seconds=10 --failure-threshold=3
----

[%collapsible]
.Expected output
====
----
deployment.apps/datasphere-api probes updated
----
====

Add a readiness probe. This tells the Service to stop sending traffic to a pod that
isn't ready:

[source,role="execute"]
----
oc set probe deployment/datasphere-api --readiness --get-url=http://:8080/health --initial-delay-seconds=5 --period-seconds=5
----

[%collapsible]
.Expected output
====
----
deployment.apps/datasphere-api probes updated
----
====

Liveness = is the app alive? If not, restart the container. Readiness = can it handle
traffic right now? If not, stop sending requests until it recovers. Both check the same
`/health` endpoint here, but they serve different purposes.

==== Explore probe settings

Look at the probe configuration:

[source,role="execute"]
----
oc describe deployment datasphere-api
----

[%collapsible]
.Expected output (key fields)
====
----
...
    Liveness:   http-get http://:8080/health delay=10s timeout=1s period=10s #success=1 #failure=3
    Readiness:  http-get http://:8080/health delay=5s timeout=1s period=5s #success=1 #failure=3
...
----
====

A few settings worth knowing:

* **`initialDelaySeconds`**: How long to wait before the first check. This gives the
  app time to start up. If your app takes 30 seconds to load a cache on startup,
  set this to 35: too short and the probe kills the app before it finishes booting,
  creating a restart loop.

* **`failureThreshold`**: How many consecutive failures before action. We set 3 for
  the liveness probe. Setting 1 is too aggressive: a single slow response would kill
  the pod. Setting 10 is too forgiving: a genuinely broken pod would run for minutes
  before being restarted.

* **`timeoutSeconds`**: How long to wait for a probe response before counting it as
  a failure. The default is 1 second. Increase this for apps with slow health check
  endpoints or endpoints that depend on downstream services.

== Module summary

In this module, you proved that:

* A single replica means any pod failure causes user-visible downtime
* Multiple replicas let the Service route around failures instantly
* The Horizontal Pod Autoscaler adds and removes pods based on actual CPU demand: no
  manual scaling needed
* Liveness probes detect unhealthy applications and restart them automatically
* Readiness probes keep traffic away from pods that aren't ready to serve
* All of this is built into the platform: no custom code, no external tools

Your DataSphere demo app can now survive pod crashes and traffic spikes. OpenShift
handles the recovery, so you don't have to.
