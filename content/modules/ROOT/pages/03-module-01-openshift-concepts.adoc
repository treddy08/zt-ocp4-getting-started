= Module 1: OpenShift Concepts -- Your First Pod
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Your team at CloudCo is ready to start building. The application is containerized and the OpenShift
cluster is running -- all that's standing between your team and a working development environment is
a deployment.

In this module, you'll first see the finished result -- a running, publicly accessible application --
and then tear it down and rebuild it yourself from scratch using the `oc` CLI.

By the end, you'll understand the three building blocks that make every OpenShift application work:
*Pods*, *Services*, and *Routes*.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application using the `oc` CLI
* Expose a deployment internally with a Service and externally with a Route
* Verify application health using `oc` commands and live HTTP requests

== Section 1: The wow moment (3 minutes)

A Hello World application is already running in your `{user}-lab-hello-world` namespace.
It was deployed automatically when your lab environment was provisioned.
Before you build anything, let's see what the finished result looks like.

=== Step 1.1: Switch to your pre-deployed namespace

[source,bash,subs="attributes+"]
----
oc project {user}-lab-hello-world
----

=== Step 1.2: See what's running

[source,bash]
----
oc get all
----

You'll see several resources: a Deployment, a Pod, a Service, and a Route.
These resources work together to run your application and expose it to the internet.

=== Step 1.3: Get the application URL

[source,bash]
----
oc get route hello-world -o jsonpath='{.spec.host}{"\n"}'
----

Copy that URL and open it in your browser (make sure to use `https://`).

You'll see a Red Hat-styled web page displaying a welcome message, along with configuration
details like the cluster domain and image version -- all values that were passed from the
catalog at deployment time.

That's a fully configured containerized web application, running on OpenShift, accessible
from anywhere on the internet -- provisioned automatically before you even logged in.

Now let's understand what made that possible.

== Section 2: Explore the pieces (5 minutes)

Every application on OpenShift is made up of the same three core resources.
Let's inspect each one and understand what it does.

=== The Pod: where your application runs

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives them
a shared network and storage context.

List the pods in your namespace:

[source,bash]
----
oc get pods
----

Expected output:
----
NAME                         READY   STATUS    RESTARTS   AGE
hello-world-xxxxxxxxx-xxxxx  1/1     Running   0          5m
----

Inspect the pod in detail:

[source,bash]
----
oc describe pod -l app=hello-world
----

Check the application logs:

[source,bash]
----
oc logs -l app=hello-world
----

NOTE: Pods are ephemeral -- they can be created, destroyed, and replaced at any time.
A *Deployment* manages this lifecycle and ensures the right number of pods are always running.

=== The Service: internal network access

A *Service* gives your application a stable internal network address. Even if the Pod restarts and
gets a new IP address, the Service address stays the same.

Inspect the Service:

[source,bash]
----
oc get svc hello-world
----

[source,bash]
----
oc describe svc hello-world
----

Notice the `Selector` field -- it uses a *label* (`app=hello-world`) to identify which pods the
Service should route traffic to. This label-based selection is how OpenShift connects resources together.

=== The Route: external access

A *Route* exposes a Service to traffic from outside the cluster. It provides a public hostname and
handles TLS termination so your users can access the app over HTTPS.

Inspect the Route:

[source,bash]
----
oc get route hello-world
----

[source,bash]
----
oc describe route hello-world
----

Notice the `Host` field -- that's the public URL you opened in your browser.
Notice the `TLS` section -- OpenShift handles the certificate automatically with edge termination.

=== How they connect

[source]
----
Browser → Route (public URL + TLS) → Service (stable internal IP) → Pod (running container)
----

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to (via labels).
The Pod runs your container.

== Section 3: Build it yourself (10 minutes)

Now that you understand the pieces, let's build it from scratch.
Switch to your clean lab namespace:

[source,bash,subs="attributes+"]
----
oc project {user}-lab
----

Verify it's empty:

[source,bash]
----
oc get all
----

Expected output:
[subs="attributes+"]
----
No resources found in {user}-lab namespace.
----

=== Step 3.1: Deploy the application

Create a Deployment from a container image:

[source,bash]
----
oc create deployment my-app \
  --image=quay.io/openshiftlabs/hello-openshift:1.0.0
----

Wait for the pod to come up:

[source,bash]
----
oc rollout status deployment/my-app
----

Expected output:
----
deployment "my-app" successfully rolled out
----

=== Verify

[source,bash]
----
oc get pods
----

Expected output:
----
NAME                      READY   STATUS    RESTARTS   AGE
my-app-xxxxxxxxx-xxxxx   1/1     Running   0          30s
----

* STATUS must be `Running`
* READY must show `1/1`

=== Step 3.2: Expose the deployment as a Service

The pod is running, but there's no stable internal address yet. Create a Service:

[source,bash]
----
oc expose deployment my-app --port=8080
----

=== Verify

[source,bash]
----
oc get svc my-app
----

Expected output:
----
NAME     TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
my-app   ClusterIP   172.30.x.x      <none>        8080/TCP   10s
----

The `ClusterIP` is the stable internal address. Other applications inside the cluster can reach
your app at this address -- but it's not yet accessible from outside.

=== Step 3.3: Create a Route for external access

Expose the Service with a Route:

[source,bash]
----
oc expose svc my-app
----

=== Verify

[source,bash]
----
oc get route my-app
----

Test it with `curl`:

[source,bash]
----
curl -s https://$(oc get route my-app -o jsonpath='{.spec.host}')
----

Expected output:
----
Hello OpenShift!
----

=== Step 3.4: Add labels to organize your resources

Labels are key/value pairs you attach to resources. They let you group, filter, and select resources
across your namespace. The Service you created uses a label selector to find its pods -- OpenShift
set this up automatically, but you can add your own labels too.

Add an environment label to your deployment:

[source,bash]
----
oc label deployment my-app environment=dev team=cloudco
----

Verify the labels were applied:

[source,bash]
----
oc get deployment my-app --show-labels
----

Now you can use label selectors to query resources:

[source,bash]
----
oc get all -l app=my-app
----

=== Final verification

Run this to see your complete application stack:

[source,bash]
----
oc get all -l app=my-app
----

You should see your Deployment, ReplicaSet, Pod, Service, and Route -- all connected by the
`app=my-app` label.

== Module summary

You've deployed a complete, publicly accessible web application on OpenShift using only the CLI.

*What you accomplished:*

* Explored a pre-deployed application and identified its Pods, Service, and Route
* Understood how the three core resources connect to serve external traffic
* Deployed a container image to OpenShift with `oc create deployment`
* Created a Service to give your app a stable internal network address
* Created a Route to expose your app to the internet with HTTPS
* Applied labels to organize and query your resources

*Key concepts:*

* *Pod* -- the running container; ephemeral and managed by a Deployment
* *Service* -- stable internal network address; uses label selectors to find pods
* *Route* -- external access with a public hostname and TLS termination
* *Labels* -- key/value metadata that connects resources together
