= Module 1: OpenShift Concepts -- Your First Deployment
:source-highlighter: rouge
:toc: macro
:toclevels: 2

In this module, you'll deploy the DataSphere frontend on OpenShift and learn the 3 building
blocks that make every OpenShift application work: *Pods*, *Services*, and *Routes*.

First you'll see the finished result — the full DataSphere application already running, live map
and all. Then you'll explore what OpenShift created to make it work. Finally, you'll deploy
the frontend yourself from scratch.

By the end, the map will be empty. That's intentional. Lab 2 picks up here.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application using the `oc` CLI
* Expose a deployment internally with a Service and externally with a Route
* Verify application health using `oc` commands

== Section 2: Explore the pieces (5 minutes)

Every application on OpenShift is built from the same 3 core resources.
Let's inspect each one using the `datasphere-ui` component.

=== The Pod: where your application runs

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives
them a shared network and storage context.

List the pods in your Project:

[source,role="execute"]
----
oc get pods
----

[%collapsible]
.Expected output
====
----
NAME                              READY   STATUS    RESTARTS   AGE
datasphere-api-7d6f9b8c4-xk2pq   1/1     Running   0          5m
datasphere-ui-5c8b4f6d9-n7rlv    1/1     Running   0          5m
----
====

Inspect the frontend pod in detail:

[source,role="execute"]
----
oc describe pod -l app=datasphere-ui
----

[%collapsible]
.Expected output (key fields)
====
----
Name:         datasphere-ui-5c8b4f6d9-n7rlv
Namespace:    {user}-datasphere
Labels:       app=datasphere-ui
Containers:
  datasphere-ui:
    Image:    quay.io/nstephan/datasphere-ui:1.0.2
    Port:     8080/TCP
    State:    Running
    Ready:    True
Conditions:
  Ready:      True
----
====

Check the application logs:

[source,role="execute"]
----
oc logs -l app=datasphere-ui
----

[%collapsible]
.Expected output (truncated)
====
----
10.x.x.x - - [31/Mar/2026:12:00:00 +0000] "GET / HTTP/1.1" 200 ...
10.x.x.x - - [31/Mar/2026:12:00:00 +0000] "GET /assets/index.js HTTP/1.1" 200 ...
----
====

NOTE: Pods are ephemeral. They can be created, destroyed, and replaced at any time.
A *Deployment* manages this lifecycle and ensures the right number of pods are always running.

=== The Service: internal network access

A *Service* gives your application a stable internal network address. Even if the Pod restarts
and gets a new IP address, the Service address stays the same.

[source,role="execute"]
----
oc get service datasphere-ui
----

[%collapsible]
.Expected output
====
----
NAME            TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
datasphere-ui   ClusterIP   172.30.198.47   <none>        8080/TCP   5m
----
====

[source,role="execute"]
----
oc describe service datasphere-ui
----

[%collapsible]
.Expected output (key fields)
====
----
Name:              datasphere-ui
Selector:          app=datasphere-ui
Type:              ClusterIP
Port:              8080/TCP
Endpoints:         10.x.x.x:8080
----
====

Notice the `Selector` field: it uses a label (`app=datasphere-ui`) to identify which pods to
route traffic to. This label-based selection is how OpenShift connects resources together.

=== The Route: external access

A *Route* exposes a Service to traffic from outside the cluster. It provides a public hostname
and handles TLS termination so your users can access the app over HTTPS.

[source,role="execute"]
----
oc describe route datasphere-ui
----

[%collapsible]
.Expected output (key fields)
====
----
Name:            datasphere-ui
Host:            datasphere-ui-{user}-datasphere.apps.{cluster_subdomain}
TLS Termination: edge
Insecure Policy: Redirect
Service:         datasphere-ui
Port:            8080
----
====

Notice the `Host` field: that's the public URL you opened in your browser.
Notice the `TLS` section: OpenShift handles the certificate automatically.

=== Why datasphere-api has no Route

Look back at the `oc get all` output from Step 1.2. `datasphere-api` has a Service but no Route.
That's deliberate: the API only needs to be reachable from within the cluster, not from the internet.

[source,role="execute"]
----
oc get service datasphere-api
----

[%collapsible]
.Expected output
====
----
NAME             TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)    AGE
datasphere-api   ClusterIP   172.30.14.22   <none>        8080/TCP   5m
----
====

The UI talks to the API by Service name — `datasphere-api` — and OpenShift's internal DNS
resolves it automatically. No Route needed.

=== How they connect

[mermaid]
....
flowchart TB
    B["🌐 Browser"]
    R["Route\n(datasphere-ui)"]
    S["Service\n(stable internal IP)"]
    P["Pod\n(running container)"]

    B -->|"HTTPS"| R
    R --> S
    S -->|"label selector"| P
....

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to via label selectors.
The Pod runs your container.

== Section 3: Build it yourself (8 minutes)

Now that you understand the pieces, let's build the frontend from scratch.
Switch to your clean lab Project:

[source,role="execute",subs="attributes+"]
----
oc project {user}-lab
----

[%collapsible]
.Expected output
====
[subs="attributes+"]
----
Now using project "{user}-lab" on server "{openshift_api_url}".
----
====

=== Step 3.1: Deploy the application

Deploy the DataSphere UI from its container image:

[source,role="execute",subs="attributes+"]
----
oc new-app {datasphere_ui_image} --name=datasphere-ui
----

[%collapsible]
.Expected output
====
----
--> Found container image ... ({datasphere_ui_image})
    ...
--> Creating resources ...
    imagestream.image.openshift.io "datasphere-ui" created
    deployment.apps "datasphere-ui" created
    service "datasphere-ui" created
--> Success
    Application is not exposed. You can expose services to the outside world by executing one or more of the commands below:
     'oc create route edge --service=datasphere-ui'
    Run 'oc status' to view your app.
----
====

`oc new-app` created 3 resources: an ImageStream to track the image version, a Deployment to
manage your pods, and a Service for internal network access. The output even tells you the next
step — the app isn't externally accessible yet.

Wait for the pod to come up:

[source,role="execute"]
----
oc rollout status deployment/datasphere-ui
----

[%collapsible]
.Expected output
====
----
deployment "datasphere-ui" successfully rolled out
----
====

NOTE: This same rollout process happens every time you deploy a new version of your application.
OpenShift creates new pods first, waits for them to be healthy, then scales down the old ones:
no downtime, no manual coordination.

=== Step 3.2: Expose the deployment as a Service

A Service already exists — `oc new-app` created it. You can verify:

[source,role="execute"]
----
oc get service datasphere-ui
----

[%collapsible]
.Expected output
====
----
NAME            TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
datasphere-ui   ClusterIP   172.30.x.x      <none>        8080/TCP   30s
----
====

The `ClusterIP` is the stable internal address. Other applications inside the cluster can reach
your app at this address, but it's not yet accessible from outside.

=== Step 3.3: Create a Route for external access

[source,role="execute"]
----
oc create route edge --service=datasphere-ui --insecure-policy=Redirect
----

[%collapsible]
.Expected output
====
----
route.route.openshift.io/datasphere-ui created
----
====

=== Verify

[source,role="execute"]
----
oc get route datasphere-ui -o jsonpath='{.spec.host}'
----

Open that URL in your browser.

The DataSphere UI loads — but the map is empty. No datacenter sites, no status data.

That's not a bug. The UI is running exactly as it should: it's already calling out to
`datasphere-api:8080`, looking for the backend. But there's no pod answering that address yet.
You've deployed the frontend. The backend doesn't exist in this Project.

In Lab 2, you'll add the API using *Source-to-Image* — OpenShift will build it directly from
source code — and the moment it comes online, the map will populate automatically. No changes
to the UI required.

== Module summary

You've deployed the frontend tier of a real 2-tier application on OpenShift.

*What you accomplished:*

* Explored a pre-deployed application and identified its Pods, Services, and Routes
* Understood how the 3 core resources connect to serve external traffic
* Understood why some components have Routes and others don't
* Deployed a container image to OpenShift with `oc new-app`
* Created a Route to expose your app to the internet with automatic HTTPS

*Key concepts:*

* *Pod*: the running container; ephemeral and managed by a Deployment
* *Service*: stable internal network address; uses label selectors to find pods
* *Route*: external access with a public hostname and automatic TLS termination
* *Labels*: key/value metadata that connects resources together

*Next steps:*

The frontend is running but the map is empty. Lab 2 adds the backend using Source-to-Image,
then shows how to deploy the complete 2-tier application as a single Helm chart.
If you already know Pods, Services, and Routes, you can start Lab 2 directly.
