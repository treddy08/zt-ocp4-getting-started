= Module 1: OpenShift Concepts -- Your First Pod
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Your team at CloudCo is ready to start building. The application is containerized and the OpenShift
cluster is running. All that's standing between your team and a working development environment is
a deployment.

In this module, you'll first see the finished result: a running, publicly accessible application.
Then tear it down and rebuild it from scratch using the `oc` CLI or the web console.

By the end, you'll understand the 3 core building blocks that make every OpenShift application work:
*Pods*, *Services*, and *Routes*.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Identify and describe the role of Pods, Services, and Routes in OpenShift
* Deploy a containerized application using the `oc` CLI or the web console
* Expose a deployment internally with a Service and externally with a Route
* Verify application health using `oc` commands or the web console

== Section 2: Explore the pieces (5 minutes)

Every application on OpenShift is made up of the same 3 core resources.
Let's inspect each one and understand what it does.

=== The Pod: where your application runs

A *Pod* is the smallest deployable unit in OpenShift. It wraps one or more containers and gives them
a shared network and storage context.

[tabs]
====
Terminal::
+
--
List the pods in your namespace:

[source,role="execute"]
----
oc get pods
----

[%collapsible]
.Expected output
=====
----
NAME                           READY   STATUS    RESTARTS   AGE
hello-world-78c55b56b9-x76k8   1/1     Running   0          7m42s
----
=====

Inspect the pod in detail:

[source,role="execute"]
----
oc describe pod -l app=hello-world
----

[%collapsible]
.Expected output (key fields)
=====
----
Name:         hello-world-78c55b56b9-x76k8
Namespace:    user-xxx-lab-hello-world
Labels:       app=hello-world
              pod-template-hash=78c55b56b9
Containers:
  hello-world:
    Image:    registry.access.redhat.com/ubi8/httpd-24:latest
    Port:     8080/TCP
    State:    Running
    Ready:    True
Conditions:
  Ready:      True
----
=====

Check the application logs:

[source,role="execute"]
----
oc logs -l app=hello-world
----

[%collapsible]
.Expected output (truncated)
=====
----
AH00558: httpd: Could not reliably determine the server's fully qualified domain name
[...] Apache/2.4 configured -- resuming normal operations
[...] Server built: ...
----
=====
--

NOTE: Pods are ephemeral. They can be created, destroyed, and replaced at any time.
A *Deployment* manages this lifecycle and ensures the right number of pods are always running.

=== The Service: internal network access

A *Service* gives your application a stable internal network address. Even if the Pod restarts and
gets a new IP address, the Service address stays the same.

[tabs]
====
Terminal::
+
--
Inspect the Service:

[source,role="execute"]
----
oc get service hello-world
----

[%collapsible]
.Expected output
=====
----
NAME          TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)    AGE
hello-world   ClusterIP   172.x.x.x      <none>        8080/TCP   8m9s
----
=====

[source,role="execute"]
----
oc describe service hello-world
----

[%collapsible]
.Expected output
=====
----
Name:              hello-world
Selector:          app=hello-world
Type:              ClusterIP
Port:              http  8080/TCP
Endpoints:         10.x.x.x:8080
----
=====
--

Console::
+
--
. In the left navigation, click *Networking* -> *Services*, then click *hello-world*.
+
The Service details page shows how internal networking works:
+
* *Pod selector*: `app=hello-world`: this label determines which pods receive traffic from this Service
* *Service address*: A *Cluster IP* (accessible within the cluster only)
* *Hostname*: The internal DNS name other applications can use to reach this service
* *Service port mapping*: Port *8080/TCP*: matching the container's listening port

image::console-guide/03-module-01-openshift-concepts/08-service-details.png[Service details showing Pod selector, Cluster IP, and port mapping]
--
====

Notice the `Selector`/`Pod selector` field: it uses a *label* (`app=hello-world`) to identify which pods the
Service should route traffic to. This label-based selection is how OpenShift connects resources together.

=== The Route: external access

A *Route* exposes a Service to traffic from outside the cluster. It provides a public hostname and
handles TLS termination so your users can access the app over HTTPS.

[tabs]
====
Terminal::
+
--
Inspect the Route:

[source,role="execute"]
----
oc get route hello-world
----

[%collapsible]
.Expected output
=====
----
NAME          HOST/PORT                             PATH   SERVICES      PORT   TERMINATION     WILDCARD
hello-world   hello-world-xxx.apps.example.com             hello-world   http   edge/Redirect   None
----
=====

[source,role="execute"]
----
oc describe route hello-world
----

[%collapsible]
.Expected output (key fields)
=====
----
Name:           hello-world
Namespace:      user-xxx-lab-hello-world
Host:           hello-world-xxx.apps.example.com
Path:           <none>
TLS Termination: edge
Insecure Policy: Redirect
Service:        hello-world
Port:           http
----
=====

Notice the `Host` field: that's the public URL you opened in your browser.
Notice the `TLS` section: OpenShift handles the certificate automatically with edge termination.
--

Console::
+
--
. Navigate to *Networking* -> *Routes* and click on *hello-world*.
+
The Route details page shows how external access works:
+
* *Location*: The public URL users can access from anywhere on the internet
* *Host*: The hostname that OpenShift generated automatically
* *Service*: Links to the *hello-world* Service, which the Route forwards traffic to
* *Target port*: `http`: the named port on the Service
* *Status*: *Accepted*: the Route is active and serving traffic

image::console-guide/03-module-01-openshift-concepts/09-route-details.png[Route details showing public URL, linked service, and target port]
--
====

=== How they connect

[mermaid]
....
flowchart LR
    B["Browser"] -->|"HTTPS"| R["Route\n(public hostname + TLS)"]
    R --> S["Service\n(stable internal IP)"]
    S -->|"label selector"| P["Pod\n(running container)"]
....

The Route knows which Service to send traffic to.
The Service knows which Pods to forward requests to (via labels).
The Pod runs your container.

=== Step 3.1: Deploy the application

[tabs]
====
Terminal::
+
--
Create a Deployment from a container image:

[source,role="execute"]
----
oc create deployment hello-openshift \
  --image=docker.io/openshift/hello-openshift
----

[%collapsible]
.Expected output
=====
----
deployment.apps/hello-openshift created
----
=====

NOTE: You may also come across `oc new-app` in OpenShift documentation. It's a higher-level command
that can auto-detect your source and create several resources at once. Here we use `oc create deployment`
because it's explicit: you'll see exactly what gets created and why.

Wait for the pod to come up:

[source,role="execute"]
----
oc rollout status deployment/hello-openshift
----

[%collapsible]
.Expected output
=====
----
deployment "hello-openshift" successfully rolled out
----
=====

NOTE: This same rollout process happens every time you deploy a new version of your application.
OpenShift creates new Pods first, waits for them to be healthy, then scales down the old ones:
no downtime, no manual coordination.
--

+
.. Enter `hello-openshift` in the *Name* field

+

You will be taken to the Deployment details page for `hello-openshift`. The pod ring shows your pods starting up.

=== Step 3.2: Expose the deployment as a Service

The pod is running, but there's no stable internal address yet. Create a Service:

[tabs]
====
Terminal::
+
--
[source,role="execute"]
----
oc expose deployment hello-openshift --port=8080
----

[%collapsible]
.Expected output
=====
----
service/hello-openshift exposed
----
=====

Verify:

[source,role="execute"]
----
oc get service hello-openshift
----

[%collapsible]
.Expected output
=====
----
NAME              TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)    AGE
hello-openshift   ClusterIP   172.x.x.x      <none>        8080/TCP   5s
----
=====
--

Console::
+
--
. In the left navigation menu, click *Networking* -> *Services*.
+
You'll see an empty list since no Services exist yet in this project.

. Click the *Create Service* button, or navigate to the *Import YAML* page using the *+* icon in the top navigation bar.

. In the YAML editor, replace any existing content with the following Service definition:
+
[source,yaml,subs="attributes+"]
----
apiVersion: v1
kind: Service
metadata:
  name: hello-openshift
  namespace: {user}-lab
spec:
  selector:
    app: hello-openshift
  ports:
    - protocol: TCP
      port: 8080
      targetPort: 8080
----

. Click *Create*.

The Service details page shows your new `hello-openshift` Service with:

* *Pod selector*: `app=hello-openshift`: this label matches the pods created by your Deployment
* *Type*: ClusterIP: a stable internal address
* *Service port mapping*: port 8080/TCP forwarding to pod port 8080

image::console-guide/03-module-01-openshift-concepts/18-service-created-details.png[Service details page showing hello-openshift Service]
--
====

The `ClusterIP` is the stable internal address. Other applications inside the cluster can reach
your app at this address, but it's not yet accessible from outside.

=== Step 3.3: Create a Route for external access

[tabs]
====
Terminal::
+
--
Expose the Service with a Route:

[source,role="execute"]
----
oc expose service hello-openshift
----

[%collapsible]
.Expected output
=====
----
route.route.openshift.io/hello-openshift exposed
----
=====

Verify:

[source,role="execute"]
----
oc get route hello-openshift
----

[%collapsible]
.Expected output
=====
----
NAME              HOST/PORT                                    PATH   SERVICES          PORT   TERMINATION   WILDCARD
hello-openshift   hello-openshift-xxx-lab.apps.example.com           hello-openshift   8080                 None
----
=====
--

Console::
+
--
. In the left navigation menu, click *Networking* -> *Routes*.
+
You'll see an empty list since no Routes exist yet.
+
image::console-guide/03-module-01-openshift-concepts/19-routes-empty_with_arrows.png[Routes page showing no routes]

. Click the *Create Route* button.

. The YAML editor opens with a default Route template. Replace the content with the following Route definition:
+
[source,yaml,subs="attributes+"]
----
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: hello-openshift
  namespace: {user}-lab
spec:
  to:
    kind: Service
    name: hello-openshift
  port:
    targetPort: 8080
----

. Click *Create*.

The Route details page confirms your Route is active:

* *Status*: Accepted
* *Location*: a public URL for your application
* *Service*: hello-openshift
* *Target port*: 8080

image::console-guide/03-module-01-openshift-concepts/21-route-created-details.png[Route details page showing hello-openshift Route with public URL]
--
====

== Module summary

You've deployed a complete, publicly accessible web application on OpenShift.

*What you accomplished:*

* Explored a pre-deployed application and identified its Pods, Service, and Route
* Understood how the 3 core resources connect to serve external traffic
* Deployed a container image to OpenShift

* Created a Route to expose your app to the internet

*Key concepts:*

* *Pod*: the running container; ephemeral and managed by a Deployment
* *Service*: stable internal network address; uses label selectors to find pods
* *Route*: external access with a public hostname and TLS termination
* *Labels*: key/value metadata that connects resources together
