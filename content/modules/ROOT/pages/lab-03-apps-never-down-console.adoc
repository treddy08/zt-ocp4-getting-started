= Module 3: Apps That Never Go Down (Web Console)
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Your DataSphere demo app is running. Users are happy. But what happens when a pod crashes?
What happens when traffic spikes at 3am? In this module, you'll find out by breaking things
on purpose, then watching OpenShift fix them automatically — all from the web console.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Explain how ReplicaSets maintain pod availability
* Configure a Horizontal Pod Autoscaler (HPA) for automatic scaling
* Describe the difference between liveness and readiness probes
* Demonstrate zero-downtime resilience under pod failure and traffic load

== Section 1: What happens when things break?

The DataSphere demo app is already deployed with resilience features configured. In this
section, you'll interact with those features by breaking things and watching OpenShift respond.

=== Kill a pod

In the left navigation, click *Workloads* → *Pods*.

You will see 3 pods running — the API, Redis, and UI. Note the *Restarts* column is 0 for
all of them.

image::console-guide/lab-03-apps-never-down/03-pods-list-all-running.png[Pods list showing all 3 pods running with 0 restarts]

Click the name of the `datasphere-ui-*` pod to open its detail page.

image::console-guide/lab-03-apps-never-down/04-ui-pod-details.png[Pod detail page showing Running status and the Actions button]

Click *Actions* → *Delete Pod*.

image::console-guide/lab-03-apps-never-down/05-actions-delete-pod.png[Actions dropdown showing the Delete Pod option]

A confirmation dialog appears showing the exact pod name and namespace.

image::console-guide/lab-03-apps-never-down/06-delete-pod-confirm.png[Delete Pod confirmation dialog]

Click *Delete*. Then switch to your browser tab with the DataSphere app — the page will
go blank as your only UI pod is gone. Within a few seconds, OpenShift creates a replacement
and the page comes back.

Switch back to the Pods list (*Workloads* → *Pods*). The console auto-refreshes — you will
see the old pod in *Terminating* status while OpenShift has already created a replacement
in *Container Creating*.

image::console-guide/lab-03-apps-never-down/07-pod-terminating-new-creating.png[Pods list showing old pod terminating and new pod already creating]

Within a few seconds the replacement pod reaches *Running* and the DataSphere app comes back.
OpenShift replaced the pod automatically — but your users experienced a real outage during
the gap. With only 1 replica, any pod failure is user-visible.

=== Fix it with replicas

In the left navigation, click *Workloads* → *Deployments*, then click *datasphere-ui*.

On the deployment detail page, click *Actions* → *Edit Pod count*.

image::console-guide/lab-03-apps-never-down/09-deployment-actions-menu.png[Deployment Actions menu showing Edit Pod count and other options]

In the dialog, change the value to `3` and click *Save*.

image::console-guide/lab-03-apps-never-down/11-edit-pod-count-3.png[Edit Pod count dialog set to 3]

Click the *Pods* tab to watch all 3 replicas come up. Wait until all show `1/1` in the
*Ready* column before continuing.

image::console-guide/lab-03-apps-never-down/12-deployment-3-pods-running.png[Pods tab showing all 3 datasphere-ui pods running 1/1]

Now delete one pod the same way you did in the previous step: go to *Workloads* → *Pods*,
click the `datasphere-ui-*` pod, then *Actions* → *Delete Pod* → *Delete*.

Switch to the browser. This time: no outage. The map stayed live the entire time. The
Service routed traffic to the 2 surviving pods while the replacement started.

[NOTE]
====
With multiple replicas, the Service load-balances across all pods with matching labels.
Losing 1 pod out of 3 means the other 2 handle requests without interruption. This is
why production workloads always run more than 1 replica.
====

=== Auto-scaling under load

So far you've been working with the UI deployment. Now let's look at the API — that's
where the auto-scaler lives.

In the left navigation, click *Workloads* → *HorizontalPodAutoscalers*.

You will see the `datasphere-api` HPA. Click it to open the details.

image::console-guide/lab-03-apps-never-down/13-hpa-details-top.png[HPA details showing min replicas 1, max replicas 5, current replicas 1]

This page shows the *Current replicas* and *Desired replicas* in real time. Keep this
tab open — you will watch these numbers change when load climbs.

For an even more visual view, open the `datasphere-api` Deployment
(*Workloads* → *Deployments* → *datasphere-api*) and click the *Metrics* tab. The CPU
usage graph will spike visibly as load increases — this is the most impressive view.

image::console-guide/lab-03-apps-never-down/14-deployment-api-metrics.png[datasphere-api Deployment Metrics tab showing live CPU usage graph]

Switch to the DataSphere app browser tab and drag the *Simulated Traffic* slider to
about *50%*.

Switch back to the console. On the HPA details page, watch the *Current replicas* and
*Desired replicas* numbers climb as CPU crosses the 50% threshold. The *Last scale time*
field will update when scaling occurs.

On the `datasphere-api` Deployment *Metrics* tab, watch the *CPU usage* graph spike
upward — you are seeing the platform respond to real load in real time.

When you are done, drag the slider back to *0%*. Replicas will begin scaling down after
about 20 seconds.

NOTE: The console refreshes on a short interval. If you want a faster view, use the
browser refresh button while watching the HPA details page.

=== Health probe: detect and fix a sick pod

[NOTE]
====
If you also want to see what happens when a pod is running but *broken* — not crashed,
just sick — continue with this section. Otherwise, jump straight to
<<section-2-build-it-yourself,Section 2: Build it yourself>>.
====

Switch to the *Terminal* tab and run:

[source,role="execute",subs="attributes+"]
----
curl -s -X POST https://$(oc get route datasphere-ui -o jsonpath='{.spec.host}')/api/break
----

Switch back to the *OpenShift Console* tab. Go to *Workloads* → *Pods* and watch the
`datasphere-api-*` pod. Within about 40 seconds you will see two things happen in sequence:

. The pod changes to *0/1* in the *Ready* column — the readiness probe pulls it from the Service
. The *Restarts* counter increments — the liveness probe restarted the container

The DataSphere map never went down: the other replicas kept serving requests while the
broken pod was restarted.

TIP: Click on the API pod name, then click the *Events* tab to see the liveness probe
failure events that triggered the restart.

=== Add auto-scaling

First, set resource requests. In *Workloads* → *Deployments*, click *datasphere-api*, then
click *Actions* → *Edit resource limits*.

image::console-guide/lab-03-apps-never-down/19-edit-resource-limits-form.png[Edit resource limits dialog showing CPU and Memory Request/Limit fields]

Set these values:

* *CPU Request*: `25` — change the unit dropdown to `m` (millicores)
* *CPU Limit*: `100` — unit `m`
* *Memory Request*: `64` — unit `Mi`
* *Memory Limit*: `128` — unit `Mi`

Click *Save*. The HPA needs these requests to have a baseline to measure CPU utilization
against. Without resource requests, the HPA has nothing to compare against.

Now click *Actions* → *Add HorizontalPodAutoscaler*.

image::console-guide/lab-03-apps-never-down/20-add-hpa-form.png[Add HorizontalPodAutoscaler form showing Name, Minimum Pods, Maximum Pods and CPU Utilization fields]

NOTE: The warning at the top confirms why you set resource requests first — the HPA cannot
measure CPU utilization without them.

Fill in the form:

* *Name*: `datasphere-api`
* *Minimum Pods*: `2`
* *Maximum Pods*: `5`
* *CPU Utilization*: `50`

Click *Save*. Open the DataSphere app and drag the *Simulated Traffic* slider to about *50%*.
Watch the HPA details page (*Workloads* → *HorizontalPodAutoscalers* → *datasphere-api*) and
the Deployment Metrics tab (*Workloads* → *Deployments* → *datasphere-api* → *Metrics*) as
replicas scale up. Slide back to *0%* when done.

=== Add health probes

Back on the `datasphere-api` Deployment details page, click *Actions* → *Add Health Checks*.

image::console-guide/lab-03-apps-never-down/21-add-health-checks.png[Add health checks page showing Readiness, Liveness, and Startup probe sections]

The page explains the difference between probe types. Click *+ Add Liveness probe*.

image::console-guide/lab-03-apps-never-down/22-liveness-probe-form.png[Liveness probe form with HTTP GET type selected]

Fill in the liveness probe:

* *Type*: `HTTP GET`
* *Path*: `/health`
* *Port*: `8080`
* *Initial delay*: `10`
* *Period*: `10`
* *Failure threshold*: `3`

Then click *+ Add Readiness probe* and fill in:

* *Type*: `HTTP GET`
* *Path*: `/health`
* *Port*: `8080`
* *Initial delay*: `5`
* *Period*: `5`

Click *Add*. Both probes are now configured. The liveness probe will restart the container
if `/health` fails 3 times. The readiness probe will remove the pod from the Service until
it recovers.

Switch to the *Terminal* tab and break one pod's health:

[source,role="execute",subs="attributes+"]
----
curl -s -X POST https://$(oc get route datasphere-ui -o jsonpath='{.spec.host}')/api/break
----

Watch *Workloads* → *Pods* — within 40 seconds, the *Restarts* counter increments on one
API pod. OpenShift detected the unhealthy pod and restarted it automatically. No alert, no
manual intervention.

== Module summary

Imagine running 500 services in production. Pods crash at 3am. Traffic spikes every Friday
afternoon. A memory leak quietly degrades 1 pod in a 10-pod deployment.

With what you configured today, none of that requires a human. The platform detects the
crashed pod and replaces it. The HPA watches CPU climb and adds capacity before users
notice slowness. The liveness probe catches the degraded pod and restarts it before the
error rate budges. No alert fires. No one gets paged. No one has to be awake.

You configured this on 1 app. The same patterns run on every app in the cluster — and
they cost you a few clicks and about 5 minutes to set up.
